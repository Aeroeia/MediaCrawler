# 断点续爬技术总结

## 1. 设计目标与适用范围

断点续爬能力用于任务调度场景下的“中断后可恢复执行”，目标是：

- 当任务手动停止或异常退出时，保留可恢复上下文。
- 下次手动执行（run-now）自动携带恢复参数，减少重复抓取。
- 在 WebUI 中可视化展示“是否可续爬”和续爬摘要。

适用范围：

- `search`（关键词模式）
- `detail`（指定 ID/URL 模式）
- `creator`（创作者模式）

不适用范围：

- 非任务调度路径（直接 CLI 单次运行）不依赖 checkpoint 表。
- `wx`（微信公众号）平台：采用实时监听语义，不启用断点续爬。

---

## 2. 架构链路

### 2.1 运行阶段

任务运行入口由 `TaskSchedulerService._start_task()` 负责：

1. 构建命令行参数（`_build_command`）。
2. 初始化任务运行时进度状态（`_init_task_progress`）。
3. 子进程日志读取（`_read_task_output`）实时更新：
   - `_track_task_progress`：更新关键词页码/已处理 ID
   - `_is_data_hit_line`：统计“有数据写入”的命中数

### 2.2 持久化阶段

任务退出后进入 `TaskSchedulerService._wait_task_process()`：

1. 根据进度状态构建 checkpoint payload（`_build_checkpoint_payload`）。
2. 按结果分支处理：
   - 手动停止：保存 checkpoint（`_save_checkpoint`）
   - 成功命中数据：删除 checkpoint（`_delete_checkpoint`）
   - 未命中数据失败：保存 checkpoint（`_save_checkpoint`）

### 2.3 恢复阶段

手动执行 `run_now()` 时默认启用 `use_resume_checkpoint=True`：

1. 读取 `crawler_task_checkpoint`。
2. 根据任务类型生成覆盖参数（`_build_resume_override`）。
3. 用覆盖参数启动新进程（注入 `--resume_mode true` 等）。

说明：`wx` 平台不读取 checkpoint，`run-now` 始终按当前任务参数直接启动监听。

## 3. 三模式恢复策略

### 3.1 `search`

- checkpoint 字段：
  - `current_keyword`
  - `current_page`
  - `remaining_keywords`
- 恢复策略：
  - `keywords = current_keyword + remaining_keywords`
  - `start_page = current_page`
  - `resume_mode = true`

### 3.2 `detail`

- checkpoint 字段：
  - `processed_ids`
  - `remaining_ids`
- 恢复策略：
  - `specified_ids = remaining_ids`
  - `resume_mode = true`

### 3.3 `creator`

- checkpoint 字段：
  - `processed_ids`
  - `remaining_ids`
- 恢复策略：
  - `creator_ids = remaining_ids`
  - `resume_mode = true`

---

## 4. 状态机与判定规则

任务运行记录（`crawler_task_run`）与任务主表（`crawler_task`）核心规则：

- 手动停止：
  - run.status = `killed`
  - run.error_message = `stopped_by_user`
  - task.status = `paused`（若任务被禁用）或 `idle`（仍启用）
  - checkpoint：保存

- 成功（有数据命中）：
  - run.status = `success`
  - task.status = `idle`
  - success_count +1
  - checkpoint：删除

- 失败（无数据命中）：
  - run.status = `failed`
  - task.status = `error`
  - fail_count +1
  - task.last_error = `no_data_crawled (exit_code=...)`
  - checkpoint：保存

## 5. 本次自检结果与已修复项

### 5.1 自检结论

- 断点续爬主链路可用：checkpoint 构建、恢复参数映射、run-now 覆盖执行逻辑完整。
- 存在一个高风险点：`wx` store 之前缺少标准化日志，导致数据命中统计与 ID 进度识别不稳定。

### 5.2 已修复项

1. `wx` store 增加标准日志（统一前缀）：
- `update_wx_account`
- `update_wx_article`
- `update_wx_article_dynamic`
- `update_wx_article_comment`

并输出关键字段：`biz/sn/article_id/url/content_id`。

2. 调度器 ID 识别规则增强：
- `_ID_CANDIDATE_PATTERNS` 扩展支持：`biz/__biz/sn`
- 支持带 `=` 的 ID 值（兼容微信 `__biz`）

3. `wx` 平台断点行为调整：
- 不读取 checkpoint 做续爬参数覆盖。
- 任务停止/失败后不保存 checkpoint，并清理历史 `wx` checkpoint。
- WebUI 固定展示“实时监听（不续爬）”。

4. 新增断点续爬单测：
- `tests/test_task_scheduler_resume.py`
- 覆盖：
  - 三模式 payload 与 override
  - 日志驱动进度跟踪
  - 数据命中规则（含微信日志）
  - success/failed/killed 的 checkpoint 保存/清理分支（含 `wx` 不续爬分支）

## 6. 已知限制与排障手册

### 6.1 MySQL 可用性

任务调度服务依赖 MySQL。若 `localhost:3306` 不可达：

- 调度初始化会报数据库连接错误。
- 断点与任务状态无法写入数据库。

优先排查：

1. MySQL 进程是否启动。
2. `config/db_config.py` 或环境变量是否指向正确主机与端口。
3. 账户权限与目标库是否存在。

### 6.2 日志格式约束

断点进度与命中统计依赖日志文本规则：

- 建议各平台 store 统一输出 `[store.<platform>.<update_/save_*>]` 风格日志。
- 若后续改动日志格式，需同步更新 `task_scheduler_service` 正则。

### 6.3 手动停止语义

点击“停止”属于主动中断：

- 运行记录记为 `killed`，不是 `failed`。
- 系统会尝试保存当前 checkpoint 供后续“续爬执行”。

### 6.4 WebUI 观察点

在任务列表重点观察三项：

- `resume_available`
- `resume_summary`
- 操作按钮文案（`续爬执行` / `立即执行`）

若显示异常，优先检查：

1. 对应任务最近运行日志是否含标准 `store.*` 命中行。
2. `crawler_task_checkpoint` 表是否有对应 `task_id` 记录。
