## 1. 你将如何阅读本项目

### 看什么文件
- `main.py`
- `cmd_arg/arg.py`
- `config/base_config.py`
- `media_platform/xhs/core.py`
- `store/xhs/__init__.py`
- `database/db_session.py`

### 为什么看
- 这 7 个文件分别代表入口层、参数层、配置层、平台业务层、存储层、数据库层、Web 控制层。
- 你先看“主干”，再看“细节”，不会一上来就被 200+ 文件劝退。

### 看完你会什么
- 能说清楚一次抓取是如何从命令进入，到落地存储，再到 WebUI 展示。
- 知道新增功能时先改哪一层，不再“全仓库搜索然后乱改”。

建议学习路径（按顺序）：
1. 先读第 3、4 节：把主链路跑通。
2. 再读第 5~8 节：搞懂参数、平台、存储、数据库。
3. 然后读第 9~12 节：理解 Web 控制面、Dashboard、代理缓存、全局状态。
4. 最后读第 13、14 节：学会排错和二次开发。

术语小贴士：
- `async/await`：异步写法，适合大量网络 I/O（比如请求接口、读写文件）。
- `I/O`：输入输出，常见是网络请求、数据库操作、磁盘读写。

## 3. 从一条命令到一次抓取

### 看什么文件
- `main.py`
- `cmd_arg/arg.py`
- `base/base_crawler.py`
- `media_platform/xhs/core.py`

### 为什么看
- 这是项目最核心的“运行链路”，理解它之后，其他模块都能挂到这条链上。

### 看完你会什么
- 能逐步解释命令执行流程。
- 知道抓取逻辑不在 API 层，而在 `main.py + media_platform/*`。

可复制命令：

```bash
# 查看参数
uv run main.py --help

# 运行一次小红书关键词抓取
uv run main.py --platform xhs --lt qrcode --type search --keywords "编程副业"
```

端到端流程（简化版）：
1. 进入 `main.py`，执行 `main()`
2. `cmd_arg.parse_cmd()` 解析参数并覆盖 `config`
3. 如果有 `--init_db`：初始化数据库并退出
4. `CrawlerFactory` 根据 `--platform` 创建平台爬虫实例
5. `crawler.start()` 执行登录、抓取、存储
6. 执行 `async_cleanup()` 释放浏览器和连接

关键函数（输入/输出/副作用）：

| 函数 | 输入 | 输出 | 副作用 |
|---|---|---|---|
| `cmd_arg.parse_cmd()` | CLI 参数 | `SimpleNamespace` | 修改全局 `config` 值 |
| `CrawlerFactory.create_crawler()` | 平台代号（如 `xhs`） | 平台爬虫实例 | 无 |
| `crawler.start()` | 无（读取 `config`） | 无 | 发起网络请求、写文件/写库 |
| `async_cleanup()` | 无 | 无 | 关闭浏览器上下文、释放资源 |

## 4. `main.py` 深度解读

### 看什么文件
- `main.py`
- `tools/app_runner.py`

### 为什么看
- `main.py` 是“总编排器”，不负责平台细节，但决定流程如何串起来。

### 看完你会什么
- 明白 `main.py` 为什么短但关键。
- 知道中断程序（Ctrl+C）后，项目如何做优雅清理。

重点代码块说明：

1. `CrawlerFactory.CRAWLERS`  
- 作用：平台代号到爬虫类的映射（`xhs/dy/ks/...`）。
- 你新增平台时，这里必须注册。

2. `main()`  
- 先解析参数，再根据参数执行“初始化数据库”或“真正抓取”。
- 抓取后还会按条件执行：
  - Excel flush（`SAVE_DATA_OPTION=excel`）
  - 评论词云生成（`ENABLE_GET_WORDCLOUD` + JSON/JSONL）

3. `async_cleanup()`  
- 优先清浏览器（CDP 或标准 Playwright）。
- 若存储为 `db/sqlite`，再关闭数据库层资源。

4. `tools.app_runner.run()`  
- 统一处理信号（`SIGINT/SIGTERM`）。
- 第一次中断走清理，第二次中断可强制退出，防止卡死。

关键函数（输入/输出/副作用）：

| 函数 | 输入 | 输出 | 副作用 |
|---|---|---|---|
| `_flush_excel_if_needed()` | 无 | 无 | 强制落盘 Excel 缓冲数据 |
| `_generate_wordcloud_if_needed()` | 无 | 无 | 读取评论文件并生成词云图文件 |
| `run(app_main, app_cleanup, ...)` | 主流程/清理回调 | 无 | 注册信号处理、取消任务、控制退出码 |

术语小贴士：
- “优雅退出”：程序接到中断后，先清资源，再退出，避免僵尸浏览器进程。

## 5. 参数与配置如何生效

### 看什么文件
- `cmd_arg/arg.py`
- `config/base_config.py`
- `config/db_config.py`
- `config/__init__.py`

### 为什么看
- 你改了配置不生效，90% 是没搞清“谁覆盖谁”。

### 看完你会什么
- 能准确判断某个参数最终值来自 CLI、配置文件还是环境变量。

配置优先级（按类别）：

1. 抓取行为参数（平台、模式、关键词等）  
- `CLI 参数 > config/base_config.py 默认值`
- 实现位置：`cmd_arg/arg.py` 中把参数写回全局 `config`

2. 数据库连接参数（MySQL/Postgres/Mongo）  
- `环境变量 > config/db_config.py 默认值`
- 例如：`MYSQL_DB_HOST`、`MYSQL_DB_USER`、`MYSQL_DB_PWD`

常见误区：
- 误区 1：只改 `db_config.py`，但运行环境里还有旧环境变量，导致连错库。
- 误区 2：在 WebUI 里选了参数，但 CLI 又用另一套参数跑，结果不一致。
- 误区 3：把 `--save_data_option db` 当成通用数据库；这里 `db` 在项目里主要指 MySQL 兼容项。

可复制命令：

```bash
# 临时覆盖 MySQL 连接信息
export MYSQL_DB_HOST=127.0.0.1
export MYSQL_DB_PORT=3306
export MYSQL_DB_USER=root
export MYSQL_DB_PWD=123456
export MYSQL_DB_NAME=media_crawler

uv run main.py --platform xhs --lt qrcode --type search --save_data_option db
```

关键函数（输入/输出/副作用）：

| 函数 | 输入 | 输出 | 副作用 |
|---|---|---|---|
| `_coerce_enum()` | 原始配置值 | 规范化枚举值 | 非法值会打印警告并回退默认值 |
| `parse_cmd()` | CLI 参数序列 | `SimpleNamespace` | 覆盖全局 `config`，改变后续运行行为 |

## 6. 平台模块模板（以 xhs 为主线）

### 看什么文件
- `media_platform/xhs/core.py`
- `media_platform/xhs/client.py`
- `media_platform/xhs/login.py`
- `media_platform/xhs/help.py`
- `media_platform/xhs/field.py`
- `media_platform/xhs/exception.py`

### 为什么看
- 7 个平台基本都遵循这套模板：`core + client + login + help + field + exception`。
- 先吃透一套，再迁移到其他平台。

### 看完你会什么
- 知道每个平台目录该放什么。
- 能快速定位“请求报错、登录失败、字段不对”分别该改哪类文件。

模块职责：

1. `core.py`  
- 业务编排层：启动浏览器、检查登录、选择 `search/detail/creator` 模式。
- 对外最重要方法是 `start()`。

2. `client.py`  
- 请求适配层：拼 URL、签名、调用 HTTP、解析错误码。
- 小红书客户端里有签名逻辑与 `pong()` 登录态检查。

3. `login.py`  
- 登录实现层：二维码/手机号/Cookie 登录。

4. `help.py`  
- 工具与解析层：URL 解析、字段提取、辅助函数。

5. `field.py`  
- 枚举与字段常量（比如排序类型、请求参数选项）。

6. `exception.py`  
- 平台特定异常类型，方便上层区分错误原因。

关键函数（输入/输出/副作用）：

| 函数 | 输入 | 输出 | 副作用 |
|---|---|---|---|
| `XiaoHongShuCrawler.start()` | 无（读取 `config`） | 无 | 启动浏览器、登录、抓取并触发存储 |
| `XiaoHongShuCrawler.search()` | 无（读取关键词） | 无 | 设置 `source_keyword_var`，循环抓取并写入 |
| `XiaoHongShuClient.request()` | `method/url/kwargs` | JSON 或文本 | 真实发 HTTP 请求，可能抛平台异常 |
| `XiaoHongShuClient.update_cookies()` | `BrowserContext` | 无 | 更新请求头 Cookie，影响后续请求认证 |

术语小贴士：
- “工厂+模板化平台目录”是为了统一扩展方式，新增平台时不会无从下手。

## 7. 数据是怎么落盘/入库的

### 看什么文件
- `store/xhs/__init__.py`
- `store/xhs/_store_impl.py`
- `store/douyin/__init__.py`（对照）
- `tools/async_file_writer.py`

### 为什么看
- 抓取结果不是直接写数据库，而是先通过平台 `StoreFactory` 决定“写到哪里”。

### 看完你会什么
- 知道 `--save_data_option` 是如何选择后端实现的。
- 知道字段何时被补充（如 `add_ts`、`source_keyword`）。

核心机制：

1. 平台 `StoreFactory` 分发  
- 例如 `XhsStoreFactory.STORES`：`csv/db/postgres/json/jsonl/sqlite/mongodb/excel`
- `create_store()` 根据 `config.SAVE_DATA_OPTION` 返回具体类实例

2. `_store_impl.py` 多实现  
- `XhsDbStoreImplement`：异步 ORM 存库
- `XhsJsonlStoreImplement`：写 JSONL 文件
- `XhsCsvStoreImplement`：写 CSV 文件
- 其他平台同理

3. `AsyncFileWriter` 文件命名与目录  
- 路径规则：`data/<platform>/<file_type>/...`
- 文件名规则：`<crawler_type>_<item_type>_<date>.<ext>`

关键函数（输入/输出/副作用）：

| 函数 | 输入 | 输出 | 副作用 |
|---|---|---|---|
| `XhsStoreFactory.create_store()` | `SAVE_DATA_OPTION` | 存储实现实例 | 选错值会抛异常 |
| `XhsDbStoreImplement.store_content()` | 内容字典 | 无 | 新增/更新数据库记录 |
| `AsyncFileWriter.write_to_jsonl()` | `item + item_type` | 无 | 追加写入磁盘文件 |
| `update_xhs_note()` | 平台原始数据 | 无 | 标准化字段后写入存储层 |

### 为什么看
- 数据库初始化、建表、会话管理都集中在这里。

- 能独立完成 SQLite/MySQL/Postgres 的初始化与接入排查。

关键职责：

可复制命令：

```bash
# 初始化 SQLite
uv run main.py --init_db sqlite

# 初始化 MySQL
uv run main.py --init_db mysql

# 初始化 Postgres
uv run main.py --init_db postgres
```

关键函数（输入/输出/副作用）：

| 函数 | 输入 | 输出 | 副作用 |
|---|---|---|---|
| `create_database_if_not_exists()` | `db_type` | 无 | 在 MySQL/Postgres 里创建数据库 |
| `create_tables()` | `db_type` | 无 | 根据 ORM 模型建表 |
| `get_session()` | 无（读取 `config`） | `AsyncSession` 上下文 | 自动提交/回滚事务 |
| `db.init_db()` | `db_type` | 无 | 触发完整建库建表流程 |

## 10. Dashboard 模块（本仓库现状）

## 11. 代理与缓存机制

### 看什么文件
- `proxy/proxy_ip_pool.py`
- `proxy/proxy_mixin.py`
- `proxy/providers/kuaidl_proxy.py`
- `proxy/providers/wandou_http_proxy.py`
- `cache/cache_factory.py`
- `cache/local_cache.py`
- `cache/redis_cache.py`

### 为什么看
- 代理是爬虫稳定性的关键；缓存是代理复用和性能优化的关键。

### 看完你会什么
- 能配置代理池并理解“代理何时自动刷新”。
- 能区分本地缓存与 Redis 缓存分别用于什么场景。

启用条件：
- `config/base_config.py` 中：
  - `ENABLE_IP_PROXY = True`
  - `IP_PROXY_PROVIDER_NAME = "kuaidaili"` 或 `"wandouhttp"`
  - `IP_PROXY_POOL_COUNT = N`

工作机制：
1. `create_ip_pool()` 创建代理池并预加载 IP
2. 平台 `client` 继承 `ProxyRefreshMixin`
3. 每次请求前 `_refresh_proxy_if_expired()` 判断并刷新代理

关键函数（输入/输出/副作用）：

| 函数 | 输入 | 输出 | 副作用 |
|---|---|---|---|
| `ProxyIpPool.get_proxy()` | 无 | `IpInfoModel` | 可能触发代理池重载与可用性校验 |
| `ProxyIpPool.get_or_refresh_proxy()` | 可选 buffer 秒数 | `IpInfoModel` | 更新当前代理状态 |
| `ProxyRefreshMixin._refresh_proxy_if_expired()` | 无 | 无 | 更新 client 的 `self.proxy` |
| `CacheFactory.create_cache()` | `memory/redis` | 缓存实例 | 选择底层缓存实现 |

术语小贴士：
- `Mixin`：不单独使用，给其他类“混入”一段可复用能力（这里是代理自动刷新）。

## 12. 关键全局状态（ContextVar）

### 看什么文件
- `var.py`
- `media_platform/*/core.py`（`source_keyword_var.set(...)`）
- `store/*/__init__.py`（`source_keyword_var.get()`）

### 为什么看
- 这是“跨函数传递上下文”的关键机制，尤其影响关键词统计和文件命名。

### 看完你会什么
- 理解为什么 `store` 层能拿到当前关键词，但函数签名里没有显式传参。
- 知道异步场景下为何使用 `ContextVar` 而不是全局普通变量。

当前关键变量：
- `crawler_type_var`：当前抓取模式（`search/detail/creator`）
- `source_keyword_var`：当前搜索关键词（用于入库字段 `source_keyword`）
- `comment_tasks_var`：评论任务列表上下文
- `request_keyword_var`：请求关键词上下文（部分平台请求构造用）

关键函数（输入/输出/副作用）：

| 函数/操作 | 输入 | 输出 | 副作用 |
|---|---|---|---|
| `source_keyword_var.set(keyword)` | 字符串关键词 | token（内部） | 改变当前协程上下文关键词 |
| `source_keyword_var.get()` | 无 | 字符串关键词 | 读取当前协程上下文值 |

术语小贴士：
- `ContextVar`：异步安全的“上下文变量”，避免并发任务互相污染全局状态。

## 13. 新手高频报错排查

### 看什么文件
- `pyproject.toml`
- `config/base_config.py`
- `config/db_config.py`
- `api/main.py`
- `media_platform/*/login.py`

### 为什么看
- 新手最耗时的不是写代码，而是环境、登录、依赖、连接问题。

### 看完你会什么
- 能快速定位是“环境问题”还是“业务代码问题”。

高频问题与处理：

1. `uv` 拉包 403（镜像或索引问题）  
- 现象：`Failed to fetch ... 403 Forbidden`
- 处理：

```bash
# 临时指定官方源
uv sync --default-index https://pypi.org/simple

# 或运行时指定
UV_DEFAULT_INDEX=https://pypi.org/simple uv run main.py --help
```

2. Playwright 相关错误  
- 现象：浏览器驱动缺失、启动失败
- 处理：

```bash
uv run playwright install
```

3. 数据库连接失败  
- 先核对环境变量与 `config/db_config.py`
- 再独立执行初始化命令确认建表通路

```bash
uv run main.py --init_db sqlite
uv run main.py --init_db mysql
```

4. 登录态问题  
- 二维码反复失效、接口 `pong` 失败
- 处理方向：改为非 headless、检查 Cookie 是否过期、确认平台风控状态

5. 被风控或验证码阻断  
- 现象：请求成功率低、验证码频繁出现
- 处理方向：降低并发、增加间隔、启用代理池、优先 CDP 模式

## 14. 二次开发起步清单

### 看什么文件
- `main.py`
- `cmd_arg/arg.py`
- `media_platform/`
- `store/`
- `database/models.py`
- `api/schemas/crawler.py`
- `api/main.py`

### 为什么看
- 你以后改需求（新增平台、新增存储）基本都绕不开这些入口。

### 看完你会什么
- 能按照最小改动路径做扩展，而不是“改一个点炸一片”。

### A. 新增平台（最小改动路径）
1. 新建 `media_platform/<name>/`，至少实现 `core/client/login/help/field/exception`
2. 在 `main.py` 的 `CrawlerFactory.CRAWLERS` 注册新平台
3. 新建 `store/<name>/`，提供 `StoreFactory + _store_impl`
4. 补充模型（如需要）到 `database/models.py`
5. 更新 CLI 枚举：`cmd_arg/arg.py` 的 `PlatformEnum`
6. 更新 API 枚举：`api/schemas/crawler.py` 与平台选项接口（`api/main.py`）
7. 运行最小 smoke 测试

### B. 新增存储后端（最小改动路径）
1. 在各平台 `_store_impl.py` 增加实现类（继承 `AbstractStore`）
2. 在各平台 `StoreFactory.STORES` 注册新 `SAVE_DATA_OPTION`
3. 更新 CLI 与 API 的 `SaveDataOptionEnum`
4. 若是数据库型后端，补 `config/db_config.py` 与 `database/db_session.py` 连接逻辑

### C. 每次改动后的自检清单
1. 命令行参数是否还能 `--help` 正常显示
2. `uv run main.py ...` 是否能最小跑通一条链路
3. 新增字段是否影响现有存储实现
4. WebUI/API 枚举是否与 CLI 一致
5. 文档是否同步更新

可复制命令：

```bash
# 基础自检
uv run main.py --help
uv run pytest
uv run pre-commit run --all-files

# Web 侧自检
uv run uvicorn api.main:app --port 8080 --reload
```

术语小贴士：
- “工厂模式”在本项目中的实际价值：把“选择哪种实现”的判断集中到一个地方，调用方不用写大量 `if/else`。
